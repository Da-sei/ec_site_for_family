## Phase 1: 基盤整備

- [x] 1. Prisma スキーマ全面再設計とインフラ基盤構築

- [x] 1.1 Prisma スキーマを新仕様に置き換える
  - 既存スキーマ（`Users`, `Items`, `Favorite`）を廃棄し、新スキーマを定義する
  - `Users`（`id`, `accountId`@unique, `name`, `passwordHash`, `createdAt`, `updatedAt`）を定義する
  - `Groups`, `GroupMembers`（@@unique[groupId, userId]）, `InviteTokens` を定義する
  - `Categories`, `Items`, `ItemImages`, `ItemDeliveryOptions` を定義する
  - `Requests` を定義する
  - `ItemStatus`, `DeliveryMethod`, `RequestStatus` の enum を定義する
  - _Requirements: 1.1, 1.4, 1.5, 2.1, 4.1_

- [x] 1.2 マイグレーション実行と初期シードデータ作成
  - `prisma migrate dev` で新スキーマをデータベースに適用する
  - `Categories` テーブルに初期カテゴリ（家電、家具、衣類、書籍、その他）をシードする
  - _Requirements: 2.1, 3.4_

- [x] 1.3 Docker Compose に uploads ボリュームを追加する
  - `docker-compose.yml` に `uploads` ボリュームを追加し、バックエンドコンテナにマウントする
  - バックエンドコンテナで `/uploads/items/` ディレクトリが永続化されることを確認する
  - `ServeStaticModule` または `express.static` で `/uploads` を静的配信するよう設定する
  - _Requirements: 2.3_

## Phase 2: バックエンド認証基盤

- [x] 2. 認証モジュール（AuthModule）実装

- [x] 2.1 UserModule の既存コードを修正する
  - `UserEntity` から `email` フィールドを削除し、`accountId`・`name`・`passwordHash` 構成に変更する
  - `UserCommandRepository.signIn()` の戻り値を `void` から `UserEntity` に修正し、インターフェースと一致させる
  - `AppModule` に `UserModule` を登録する
  - _Requirements: 1.1, 1.4_

- [x] 2.2 AuthModule を新規作成し、JWT 認証基盤を構築する
  - `@nestjs/passport`, `passport-jwt`, `@nestjs/jwt`, `bcryptjs` パッケージを追加する
  - `AuthModule` を作成し、`JwtModule`（シークレット: 環境変数 `JWT_SECRET`、有効期限: `7d`）を設定する
  - `LocalStrategy`（accountId + パスワード検証）と `JwtStrategy`（トークン検証）を実装する
  - _Requirements: 1.2, 1.9_

- [x] 2.3 ユーザー登録エンドポイントを実装する
  - `POST /auth/register` を実装する（リクエスト: `{ name, password }`）
  - UUID v4 で `accountId` を自動生成し、`bcryptjs`（saltRounds=10）でパスワードをハッシュ化する
  - 登録成功時に `{ accountId, accessToken }` を返す
  - パスワード 8 文字未満・名前空のリクエストは 400 を返す
  - _Requirements: 1.1, 1.4_

- [x] 2.4 ログインエンドポイントを実装する
  - `POST /auth/login` を実装する（リクエスト: `{ accountId, password }`）
  - 存在しない accountId または誤ったパスワードの場合に 401 と「アカウントIDまたはパスワードが正しくありません」を返す
  - 認証成功時に `{ accessToken }` を返す
  - _Requirements: 1.2, 1.3_

- [x] 2.5 JwtAuthGuard をグローバル登録し、@Public() デコレータを実装する
  - `JwtAuthGuard` を `APP_GUARD` としてグローバルに登録する
  - `@Public()` カスタムデコレータを作成し、`/auth/register` と `/auth/login` をスキップ対象にする
  - JWT なしリクエストが保護エンドポイントへアクセスした際に 401 を返すことを確認する
  - _Requirements: 1.9_

## Phase 3: バックエンド グループ管理

- [x] 3. グループモジュール（GroupModule）実装

- [x] 3.1 GroupModule を作成し、グループ作成エンドポイントを実装する
  - `POST /groups` を実装する（リクエスト: `{ name }`）
  - 作成者を `GroupMembers` にオーナーとして自動登録する
  - 作成したグループ情報（`GroupDto`）を 201 で返す
  - _Requirements: 1.5_

- [x] 3.2 招待トークン発行エンドポイントを実装する
  - `POST /groups/:id/invite` を実装する
  - UUID v4 で招待トークンを生成し、有効期限（48時間）を設定して `InviteTokens` に保存する
  - グループオーナー以外からのリクエストは 403 を返す
  - `{ token, expiresAt }` を 201 で返す
  - _Requirements: 1.6_

- [x] 3.3 招待トークンによるグループ参加エンドポイントを実装する
  - `POST /groups/join` を実装する（リクエスト: `{ token }`）
  - トークンが期限切れまたは無効の場合は 400 と「招待リンクが無効または期限切れです」を返す
  - 既に参加済みの場合は 409 を返す
  - 参加成功時に `GroupMemberDto` を 200 で返す
  - _Requirements: 1.7, 1.8_

- [x] 3.4 自分が所属するグループ一覧取得エンドポイントを実装する
  - `GET /groups/mine` を実装し、認証ユーザーが所属するグループ一覧を返す
  - _Requirements: 1.5_

## Phase 4: バックエンド 商品管理

- [x] 4. 商品モジュール（ItemModule）実装

- [x] 4.1 ItemModule を作成し、商品出品エンドポイントを実装する
  - `POST /items` を実装する（リクエスト: `CreateItemDto`）
  - タイトル必須・200文字以内、渡し方オプション 1件以上を検証し、違反時に 400 を返す
  - `Items`, `ItemDeliveryOptions` を作成し、ステータスを `AVAILABLE` で保存する
  - 作成した商品の詳細（`ItemDetailDto`）を 201 で返す
  - _Requirements: 2.1, 2.2, 2.7_

- [x] 4.2 GroupMemberGuard を実装し、グループスコープを適用する
  - `GroupMemberGuard` を実装する：クエリパラメータの `groupId` またはアイテム経由のグループを検証する
  - 商品一覧・詳細エンドポイントに `GroupMemberGuard` を適用する
  - グループ非メンバーのアクセスに 403 を返す
  - _Requirements: 3.1, 5.4_

- [x] 4.3 商品画像アップロードエンドポイントを実装する
  - `POST /items/:id/images` を `multer` インターセプターで実装する
  - ファイルを `/uploads/items/{uuid}.{ext}` に保存し、URL を `ItemImages` に保存する
  - ファイルサイズ上限（5MB）超過時は 400 を返す
  - 出品者以外からのアクセスは 403 を返す（`ItemOwnerGuard` を実装する）
  - _Requirements: 2.3, 2.5_

- [x] 4.4 商品一覧・詳細・検索エンドポイントを実装する
  - `GET /items` を実装する：`AVAILABLE`・`IN_TRANSACTION` ステータスの商品のみ新着順で返す
  - `keyword`（タイトル/説明の部分一致）・`categoryId` によるフィルタを実装する
  - `offset`/`limit`（最大50件）によるページネーションを実装する
  - `GET /items/:id` を実装し、詳細情報を返す（存在しない場合は 404）
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 3.5, 3.6_

- [x] 4.5 商品更新・削除エンドポイントを実装する
  - `PATCH /items/:id` を実装する：タイトル・説明・カテゴリ・渡し方の変更を反映する
  - `DELETE /items/:id` を実装する：ステータスを `DELETED` に更新する論理削除
  - 出品者以外のアクセスは 403 を返す
  - `DELETED` 商品は `GET /items` 一覧に含めない
  - _Requirements: 2.4, 2.5, 2.6_

## Phase 5: バックエンド 取引管理

- [x] 5. 取引申し込みモジュール（RequestModule）実装

- [x] 5.1 RequestModule を作成し、申し込みエンドポイントを実装する
  - `POST /requests` を実装する（リクエスト: `{ itemId }`）
  - 出品者自身の申し込みは 400 と「自分の出品には申し込みできません」を返す
  - 商品が `AVAILABLE` でない場合は 409 と「この商品はすでに取引中または譲渡済みです」を返す
  - 申し込みレコード（`PENDING`）を作成し、`RequestDto` を 201 で返す
  - _Requirements: 4.1, 4.2, 4.3, 4.6_

- [x] 5.2 申し込み承認・辞退エンドポイントを実装する
  - `POST /requests/:id/approve` を実装する：DB トランザクションで `Item.status → IN_TRANSACTION`、`Request.status → APPROVED` を原子的に更新する
  - `POST /requests/:id/decline` を実装する：`Request.status → DECLINED` に更新する
  - 出品者以外のアクセスは 403 を返す
  - `IN_TRANSACTION` 中は別ユーザーの新規申し込みを 409 で拒否する
  - _Requirements: 4.4, 4.5, 4.6_

- [x] 5.3 取引キャンセルエンドポイントを実装する
  - `POST /requests/:id/cancel` を実装する
  - 出品者または申し込み者がキャンセルを実行できる
  - `Item.status → AVAILABLE`、`Request.status → CANCELLED` に更新する
  - _Requirements: 4.7_

- [x] 5.4 取引完了エンドポイントを実装する
  - `POST /requests/:id/complete` を実装する
  - 申し込み者のみが完了報告できる（出品者は 403）
  - `Item.status → TRANSFERRED`、`Request.status → COMPLETED`、`completedAt` を記録する
  - 取引に関与していないユーザーのアクセスは 403 を返す
  - _Requirements: 5.1, 5.4_

- [x] 5.5 取引履歴・申し込み一覧エンドポイントを実装する
  - `GET /requests/history` を実装する：`COMPLETED` ステータスの取引を日付降順で返す（出品・申し込み両方）
  - `GET /requests/my-requests` を実装する：自分の申し込み一覧を返す
  - `GET /requests?itemId=:id` を実装する：商品に対する申し込み一覧を返す（出品者のみ）
  - _Requirements: 5.2, 5.3_

## Phase 6: Flutter 認証・グループ機能

- [x] 6. Flutter アプリ基盤と認証機能実装

- [x] 6.1 Flutter プロジェクト基盤を構築する

  - `flutter_secure_storage` を使用して JWT を Keychain / Keystore に保存・読み取りするリポジトリを実装する
  - JWT を全リクエストに自動付与するインターセプターを実装する
  - 401 レスポンス時にログイン画面へリダイレクトする処理を実装する
  - _Requirements: 7.3_

- [x] 6.3 ログイン・登録画面を実装する
  - ログイン画面（accountId・パスワード入力フォーム）を実装する
  - 新規登録画面（名前・パスワード入力フォーム）を実装する
  - Riverpod で認証状態（`AuthState`）を管理し、成功時に JWT をセキュアストレージに保存する
  - API エラー時に分かりやすいエラーメッセージを表示し、再試行ボタンを提供する
  - ローディング中はインジケーターを表示する
  - _Requirements: 1.1, 1.2, 7.4, 7.8, 7.9_

- [x] 6.4 グループ管理画面を実装する (P)
  - グループ一覧・作成画面を実装する
  - 招待コード発行 UI と、招待コードを入力してグループに参加する UI を実装する
  - Riverpod で `GroupState` を管理する
  - _Requirements: 1.5, 1.6, 1.7, 1.8_
